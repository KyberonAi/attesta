# Spec: TrailProof Integration

branch: feature/trailproof-integration

## Status
Approved

## Problem
Attesta ships its own hand-rolled SHA-256 hash-chained JSONL audit logger. TrailProof (`pip install trailproof` / `npm install @kyberonai/trailproof`) is a standalone, zero-dependency library purpose-built for tamper-evident audit logging with HMAC signing, multi-tenancy, trace/session correlation, and cursor-based queries. Maintaining a bespoke audit chain inside Attesta duplicates effort and misses features that TrailProof already provides.

## Goals
- Replace custom audit chain logic with TrailProof as an optional backend
- Introduce an `AuditBackend` protocol so backends are swappable
- Keep the existing `AuditLogger` fully functional as the default ("legacy") backend
- Support switching backends via config (`audit.backend: trailproof | legacy`)
- Enrich audit events with multi-tenancy (`tenant_id`) and trace correlation (`trace_id`)

## Non-Goals
- Removing or deprecating the legacy audit logger in this iteration
- Building a query UI or dashboard on top of TrailProof events
- Adding TrailProof as a required (non-optional) dependency
- Modifying the existing JSONL field order or hash chain format in the legacy backend

## Background
Attesta's current audit system (`core/audit.py`) writes `AuditEntry` dataclasses to a JSONL file with SHA-256 hash chaining. Each entry records: entry_id, intercepted_at, agent_id, session_id, action_name, risk_score, risk_level, challenge_type, verdict, challenge_duration_ms, and chain_hash. Verification walks the file and recomputes hashes.

TrailProof provides a 10-field event envelope (event_id, event_type, timestamp, actor_id, tenant_id, trace_id, session_id, payload, prev_hash, hash, signature) with HMAC signing, cursor-based pagination, and store abstraction (in-memory or JSONL). It is maintained as a sibling package in the Kyberon AI ecosystem.

The `AuditBackend` protocol formalizes what both backends share: logging an approval result and verifying integrity. This keeps the door open for future backends (database-backed, remote API, etc.) without framework coupling.

## Solution Overview
Define a minimal `AuditBackend` protocol in `core/types.py`. Wrap TrailProof in a `TrailProofBackend` adapter that maps Attesta audit fields onto TrailProof events. Register the legacy `AuditLogger` as `LegacyBackend` conforming to the same protocol. Let the config loader select the backend by name. TrailProof remains an optional extra -- importing it fails gracefully if the package is not installed.

## Design Details

### Data Model

**Field mapping (Attesta AuditEntry -> TrailProof TrailEvent):**

| Attesta field          | TrailProof field | Notes                                         |
|------------------------|------------------|-----------------------------------------------|
| entry_id               | event_id         | Auto-generated by TrailProof, not passed in   |
| (derived)              | event_type       | `"attesta.approval.{verdict}"` e.g. `"attesta.approval.approved"` |
| intercepted_at         | timestamp        | Auto-generated by TrailProof                  |
| agent_id               | actor_id         | Direct mapping                                |
| (from config)          | tenant_id        | Set once at backend init from policy/config   |
| session_id             | session_id       | Direct mapping                                |
| (not used yet)         | trace_id         | Optional, passed through ActionContext.hints   |
| action_name, risk_score, risk_level, challenge_type, challenge_duration_ms, verdict, metadata | payload | All remaining fields bundled into payload dict |
| (internal)             | prev_hash        | Managed by TrailProof                         |
| (internal)             | hash             | Managed by TrailProof                         |
| (if HMAC key set)      | signature        | Managed by TrailProof                         |

**AuditBackend protocol (new in `core/types.py`):**

Minimal surface -- just `log` and `verify`. Backends own their own construction and configuration.

### Key Logic

**Event type derivation:** The event_type is constructed as `f"attesta.approval.{result.verdict.value}"`, producing values like `"attesta.approval.approved"`, `"attesta.approval.denied"`, `"attesta.approval.timed_out"`.

**Payload construction:** All Attesta-specific audit fields that do not map to top-level TrailProof fields are bundled into the `payload` dict:
```python
payload = {
    "action_name": context.function_name,
    "risk_score": result.risk_score,
    "risk_level": result.risk_level.value,
    "challenge_type": result.challenge_type.value,
    "challenge_duration_ms": result.challenge_duration_ms,
    "metadata": result.metadata or {},
}
```

**Lazy import:** TrailProof is imported inside `TrailProofBackend.__init__()`, not at module top level. If the package is missing, raise `ImportError` with a clear message: `"trailproof is required for the TrailProof audit backend. Install with: pip install attesta[trailproof]"`.

**Config loader wiring:** The config loader reads `audit.backend` (default `"legacy"`). When set to `"trailproof"`, it instantiates `TrailProofBackend` with the remaining `audit.*` fields. The `Attesta` entry-point class accepts an optional `audit_backend: AuditBackend` parameter; if not provided, it falls back to legacy.

**Verify passthrough:** `TrailProofBackend.verify()` calls `tp.verify()` and maps the result to `AuditVerifyResult(intact=result.intact, total=result.total, broken=result.broken)`.

## Edge Cases
- TrailProof not installed and `audit.backend: trailproof` configured: raise `ImportError` at init time with install instructions, not at first log call.
- Empty audit trail: `verify()` returns `AuditVerifyResult(intact=True, total=0, broken=0)`.
- `tenant_id` not set in config: default to `"default"`.
- `agent_id` is None on ActionContext: pass `"anonymous"` as `actor_id` to TrailProof.
- `hmac_key_env` points to an unset env var: log a warning and proceed without HMAC signing (TrailProof supports unsigned events).
- Concurrent writes: TrailProof handles its own file locking; no additional locking needed in the backend adapter.
- Migration from legacy to TrailProof: audit trails are independent files. No automatic migration of existing JSONL entries. Document this.

## Open Questions
- [ ] Should we provide a CLI command to migrate legacy audit entries into TrailProof format?
- [ ] Should `trace_id` be a first-class field on `ActionContext` rather than living in `hints`?

## Out of Scope
- Query API exposure (TrailProof's `tp.query()` and `tp.get_trace()` are available directly but not surfaced through an Attesta abstraction in this iteration)
- Dashboard or analytics built on TrailProof events
- Remote/cloud audit store backends
- Automatic migration tooling for existing legacy audit files

## Acceptance Criteria
- [ ] `AuditBackend` protocol defined in `core/types.py` (Python) and `core/types.ts` (TypeScript)
- [ ] `AuditVerifyResult` dataclass/interface defined in both SDKs
- [ ] `TrailProofBackend` implemented in both Python and TypeScript SDKs
- [ ] `TrailProofBackend` passes all existing audit tests (log + verify round-trip)
- [ ] Legacy `AuditLogger` unchanged, remains the default backend, conforms to `AuditBackend` protocol
- [ ] Config loader supports `audit.backend` field with values `"trailproof"` and `"legacy"`
- [ ] TrailProof is an optional dependency (`pip install attesta[trailproof]` / no change for npm since it is listed as optional peer dep)
- [ ] Core works without TrailProof installed (no top-level import)
- [ ] `tp.verify()` is used for integrity checks when TrailProof backend is active
- [ ] Event type follows `"attesta.approval.{verdict}"` naming convention
- [ ] HMAC key is read from env var, not stored in config file

## Testing Guidelines
Create test files for the new feature and cover the following scenarios:

**Python (`tests/test_trailproof_backend.py`):**
- TrailProofBackend logs an approval result and verify returns intact=True
- Verify detects tampering (broken > 0) after manual file corruption
- Event type is correctly derived from each Verdict variant
- Payload contains all expected Attesta fields (action_name, risk_score, etc.)
- actor_id falls back to "anonymous" when agent_id is None
- tenant_id defaults to "default" when not configured
- ImportError is raised at init when trailproof package is not installed
- Backend works with both in-memory and JSONL stores
- AuditBackend protocol is satisfied by both TrailProofBackend and LegacyBackend

**TypeScript (`tests/trailproof-backend.test.ts`):**
- TrailProofBackend logs and verifies round-trip
- Event type derivation for each verdict
- Payload field mapping
- Graceful handling when @kyberonai/trailproof is not installed
- AuditBackend interface conformance for both backends
